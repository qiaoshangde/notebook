

# 数据集建议

不是不能用 `CIC-IDS2017`，而是它更适合做“辅助验证/轻量主实验”，不太适合单独当你这篇论文的唯一主数据集。

原因主要有三点。

第一，`CSE-CIC-IDS2018` 更大、更近、攻击场景更完整。官方页写得很明确：`CSE-CIC-IDS2018` 有约 `16,233,002` 条样本、`10` 天流量、`7` 类攻击场景和 `80` 个流特征；而 `CICIDS2017` 约 `2,830,743` 条样本、`5` 天流量。[UNB 2018 官方页](https://www.unb.ca/cic/datasets/ids-2018.html) 和 [CSE-CIC-IDS2018 综述](https://link.springer.com/article/10.1186/s40537-020-00382-x) 都支持这一点。对你这种“未知攻击检测 -> 聚类 -> 增量学习”的题目，主数据集更大，留出未知类后更稳，也更容易做外部泛化。

第二，`CICIDS2017` 本身有比较知名的数据质量问题。`WTMC 2021` 的官方补充页直接写到：他们在复查 `CICIDS2017` 时发现了流量生成、flow 构造、特征提取和标注问题，最终“超过 20% 的原始流量轨迹被重建或重标注”。这不是说 2017 不能用，而是说如果用它，最好用修正版，而不要把原版直接当金标准。[Troubleshooting an Intrusion Detection Dataset: the CICIDS2017 Case Study](https://intrusion-detection.distrinet-research.be/WTMC2021/index.html)

第三，开放世界/未知攻击这条线里，很多论文根本不只盯着 `CIC-IDS2017`。更近的代表工作 `Trident`（WWW 2024）做的是“未知流量检测 + 类增量更新”，但它评估用的是多套流量数据，不是只靠 `CICIDS2017`/`CSE-CIC-IDS2018` 一套吃到底，说明这个方向更看重“开放世界设定”和“跨数据集验证”，而不只是传统闭集分类 benchmark。[Trident, OpenReview](https://openreview.net/forum?id=rfXUaRlL0V)

如果你问“别人发的论文里都用什么”，可以粗分成三类：

- 普通 IDS 分类论文：`CICIDS2017` 仍然非常常见，因为它小、成熟、复现快。
- 更近、强调规模和现代流量的论文：越来越多用 `CSE-CIC-IDS2018`，尤其是大规模深度学习或综述类工作。[CSE-CIC-IDS2018 综述](https://link.springer.com/article/10.1186/s40537-020-00382-x)
- 未知攻击 / 开放世界 / 增量学习论文：常见组合不是只用 `CIC-IDS2017`，而是 `CSE-CIC-IDS2018`、`UNSW-NB15`、IoT 数据集，甚至像 `Trident` 这种直接用别的流量基准来做开放世界评估。[Trident](https://openreview.net/forum?id=rfXUaRlL0V), [UNSW-NB15 官方页](https://research.unsw.edu.au/projects/unsw-nb15-dataset)

我的建议不变，但我会稍微收紧一下：

- 主数据集用 `CSE-CIC-IDS2018`
- 外部验证用 `UNSW-NB15`
- `CIC-IDS2017` 作为辅助 sanity check
- 如果后面你想压缩工作量，也可以改成“`CIC-IDS2017` 主实验 + `CSE-CIC-IDS2018` 补验证”，这个更轻，但投稿说服力会弱一点
