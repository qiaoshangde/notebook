
# 1、为什么现在的code agent 都转向了grep，rg？

1、代码结构天然适合grep，代码中有大量的稳定符合，函数，变量，类，接口等
2、codeing agent需求精确无比的定位。（gerp可以给agent一个确定的起点）
3、ast/lsp一个函数，变量，在项目中对应一条具体的执行链路，可以顺着链路，函数在哪定义，被谁调用。可以看到整个项目的结构。

# 2、解释AST/LSP.

[深入理解LSP Understanding Language Server Protocol - 知乎](https://zhuanlan.zhihu.com/p/1890892177544021181)
**AST** 和 **LSP** 都是代码编辑器、IDE、代码分析工具里常见的东西。

**AST，抽象语法树**，可以理解为：把一段代码拆成一棵“结构树”。

比如代码：

```python
a = 1 + 2
```

人看到的是一行代码，但程序分析工具会把它理解成：

```text
赋值语句
├── 左边：变量 a
└── 右边：加法表达式
    ├── 数字 1
    └── 数字 2
```

这棵树就是 AST。它的作用是让程序“看懂代码结构”，而不是只把代码当成普通文本。比如格式化代码、检查语法错误、重构变量名、自动补全、代码生成，都可能用到 AST。

**LSP，Language Server Protocol，语言服务器协议**，可以理解为：编辑器和“代码理解服务”之间的一套通用通信规则。

以前每个编辑器都要自己支持 Python、Java、Go、Rust 等语言，成本很高。LSP 的思路是：把“理解代码”的能力单独做成一个语言服务器，编辑器只负责和它通信。

例如你在 VS Code、Cursor、Neovim 里写代码时，能看到：

```text
代码补全
跳转到定义
查找引用
悬停查看说明
语法报错提示
自动重命名
```

这些很多都是 LSP 提供的。
在传统IDE中，LSP的价值主要体现在复用语言能力和提升编辑体验上；而在AI编程工具中，LSP的价值进一步放大，尤其体现在对大模型token消耗的显著节省上。

引入LSP之后，编程语言相关的确定性计算会在本地由语言服务完成。符号表、类型信息、引用关系、诊断结果等都可以被精确计算出来，再以高度结构化、压缩后的形式提供给模型。模型接收到的不再是冗长的源码文本，而是已经被“语义提纯”的结果，本质上是用本地计算替代token消耗。

可以这样理解二者关系：

**AST 是代码的内部结构表示，LSP 是编辑器和代码分析工具之间的通信协议。**
再通俗一点：

```text
AST：把代码拆开，看懂代码本身是什么结构。
LSP：让编辑器问语言服务器：“这个变量定义在哪？”“这里有没有错误？”“帮我补全一下。”
```
举个完整例子：
你在编辑器里写：
```python
print(name)
```
编辑器通过 LSP 问 Python 语言服务器：“这里的 name 是什么？”  
语言服务器分析代码，内部可能会用 AST 等结构来理解代码，然后返回结果：这个变量在哪里定义、有没有类型错误、能不能跳转。
所以：

```text
AST 更偏底层：用于分析代码结构。
LSP 更偏工具层：用于让编辑器获得代码智能能力。
```

# 3、agent和workflow和区别
我理解的 **workflow 更强调“流程编排”**，而 **agent 更强调“自主决策和动态执行”**。

Workflow 通常是预先定义好的执行流程，比如第一步读取合同，第二步抽取关键信息，第三步调用规则引擎，第四步生成审查报告。它的路径相对固定，适合规则明确、步骤稳定、结果可控的业务场景。比如报销审批、订单处理、合同流转，这些都很适合用 workflow 来做。

Agent 则更像一个具备目标导向能力的执行单元。它不只是按照固定步骤走，而是会根据当前任务状态进行判断，决定下一步该调用什么工具、是否需要检索资料、是否需要追问用户、是否需要反复修改结果。比如让一个代码修改 agent 修复 bug，它可能先读代码，再定位错误，再运行测试，再修改文件，再根据报错继续调整，这个过程不是完全固定的，而是动态规划出来的。

所以两者最大的区别在于：**workflow 是人提前设计好的流程，agent 是模型根据目标和上下文自主选择行动路径。**

在工程实践里，我不会把它们完全对立起来。更合理的做法是把 workflow 和 agent 结合使用：用 workflow 约束大的业务流程，保证系统稳定、可观测、可恢复；在某些需要灵活判断的节点里嵌入 agent，让它完成复杂决策或非结构化任务。比如企业合同审查系统中，整体审批链路可以是 workflow，但其中“识别风险条款、判断风险等级、生成修改建议”这些环节可以交给 agent。

总结来说，**workflow 解决的是流程确定性和工程可控性问题，agent 解决的是复杂任务中的自主判断和动态执行问题。面试中如果做企业级系统，我会更倾向于用 workflow 作为骨架，用 agent 作为智能节点，而不是让 agent 完全自由运行。**


# 4、大模型是如何使用工具的。

大模型会生成，但是不会直接使用工具。
预先告诉模型有什么工具，如何使用工具。模型会根据这些，来生成它要使用的工具的名字以及参数。模型只会生成这些，而不会直接使用。收到模型生成的工具名字以及参数时，可以将其映射到一个函数。（绝大多数的工具都可以做成一个函数，负责工具的执行）。工具执行完，将结果返回大模型，就完成了一次工具的调用。
模型负责决策和生成调用参数；工具定义规定这个工具能做什么、需要什么输入、返回什么输出；执行器负责真正调用外部系统，并把结果返回给模型。
模型不直接拥有外部能力，而是通过结构化工具调用，把自己的语言理解和推理能力连接到搜索、代码、数据库、文件、业务系统等外部能力上，从而完成更复杂、更实时、更落地的任务。
在企业级场景里，工具调用还需要特别注意权限控制、参数校验、日志审计和失败重试。因为大模型可能会生成不合理的参数，所以不能让它无约束地直接操作数据库、发邮件或修改生产系统。通常会加一层安全网，比如只开放白名单 API、限制高危操作、关键动作要求人工确认。

# 5、大模型主流架构

**第一类是 ReAct，也就是推理与行动协同。**

ReAct 可以理解为很多 Agent 架构的起点。它的流程是：先思考当前问题，然后采取行动，比如调用工具、检索资料、执行代码；拿到反馈后再继续思考，再决定下一步动作，直到任务完成。

比如用户问一个需要查资料的问题，Agent 不是直接回答，而是先判断需要搜索，然后调用搜索工具，看到结果后再分析是否足够，必要时继续搜索或整理答案。

它的优点是简单、通用，很多任务都可以套这个模式。缺点是每一步都可能要反复调用大模型和工具，所以成本比较高，速度也可能比较慢。

**第二类是 Plan-and-Execute，也就是规划与执行分离。**

这个架构强调“先想清楚再动手”。它通常分成两个阶段：第一阶段先做规划，把复杂任务拆成多个子任务，并确定执行顺序；第二阶段再逐个执行，并检查执行结果。如果某一步失败，可以回退、重试或者重新规划。

比如让 Agent 分析一个代码仓库，它可以先规划：读取目录结构、分析启动入口、梳理核心模块、总结技术栈、生成改造方案。然后再按步骤执行。

它的优点是适合复杂任务，流程更清晰，也更容易工程化和回滚。缺点是规划本身不一定准确，如果一开始拆错任务，后面执行也会偏；另外对于简单任务来说，这种方式会显得太重。

**第三类是 Multi-Agent，也就是多智能体协作。**

Multi-Agent 的思想是，一个 Agent 不一定要完成所有事情，可以让多个 Agent 分工协作。通常会有一个协调器负责分配任务，下面有多个 specialist，比如搜索 Agent、代码 Agent、写作 Agent、评审 Agent、测试 Agent 等。它们可以共享知识库，最后由协调器汇总结果。

比如企业合同审查系统里，可以有合同解析 Agent、风险识别 Agent、法规检索 Agent、修改建议 Agent、报告生成 Agent。每个 Agent 专注一个环节。

它的优点是专业分工明确，复杂任务处理能力更强，整体上更像一个真实团队。缺点是协调成本高，多个 Agent 之间可能出现意见冲突、重复工作或者结果不一致，所以需要设计好通信机制、角色边界和最终仲裁逻辑。

**第四类是 Reflective Agent，也就是反思型智能体。**

Reflective Agent 的核心是让模型自己给自己做质检。它不是生成一次就结束，而是会经历“生成结果、反思检查、发现问题、修改结果、再次检查”的过程。一般不会无限循环，工程上通常会限制在 3 到 5 轮。

比如代码生成后，它可以检查是否满足需求、是否有边界情况、是否可能运行失败；论文润色后，它可以检查逻辑是否连贯、术语是否统一、是否有表达重复。

它的优点是可以提升输出质量，尤其适合代码、文档、审查报告这类对准确性要求较高的场景。缺点是成本比较高，因为每一轮反思和修改都需要再次调用大模型。

**第五类是 Tool-Augmented，也就是工具增强型 Agent。**

Tool-Augmented 可以理解为给大模型配上“手脚”。大模型本身只能生成文本，不会真正查数据库、运行代码、发邮件、读文件或者调用业务系统。所以需要给它注册一组工具，让它根据任务自己判断该调用哪个工具，并生成对应参数。

比如用户让 Agent 查询订单状态，模型会选择订单查询 API；用户让它分析 Excel，它会调用文件解析工具；用户让它运行实验，它会调用代码执行环境。

它的优点是让大模型从单纯对话变成可以真正执行任务的系统，这是现代 Agent 的基础能力。缺点是工具质量非常关键，工具接口设计不好、返回结果不稳定或者权限控制不合理，都会影响 Agent 的效果。

**第六类是 Memory-Augmented，也就是记忆增强型 Agent。**

Memory-Augmented 的核心是让 Agent 具备记忆能力，而不是每次对话都从零开始。一般可以分成几层：当前会话记忆、短期任务记忆、长期用户偏好记忆，以及向量数据库中的知识记忆。系统会根据重要性和使用频率，对记忆进行写入、更新、检索和清理。

比如一个长期代码助手可以记住项目技术栈、目录结构、接口规范和用户偏好的代码风格；一个学习助手可以记住用户正在准备什么考试、薄弱知识点是什么。

它的优点是适合长期交互和连续任务，不会聊几轮就丢失上下文。缺点是管理复杂，记忆可能过期、冲突或者膨胀，所以需要有记忆更新、清理和权限控制机制。

**第七类是 RAG Agent，也就是检索增强生成。**

RAG Agent 的核心是“先查资料再回答”。它通常会把用户问题向量化，然后去知识库或者向量数据库中检索相关内容，对结果进行排序和筛选，再把最相关的内容拼接到上下文中，最后交给大模型生成答案。

普通 RAG 更像是固定流程：检索一次，然后回答。而 RAG Agent 更灵活，它可以决定查什么关键词、是否多轮检索、是否换查询方式、是否结合工具进一步验证。

它的优点是可以减少幻觉，回答有依据，也方便接入企业内部知识库，并且知识可以实时更新。缺点是检索质量决定了生成质量，如果知识库构建不好、切分不好、召回不准，模型最后也很难答好。

**第八类是 Autonomous Loop，也就是自主智能体循环。**

Autonomous Loop 更接近“自动驾驶式”的 Agent。用户只给一个目标，Agent 会自动拆任务、排优先级、执行任务、评估结果、记录状态、追踪进度，然后继续执行下一个任务，形成持续循环。

比如让 Agent 自动修复一个 bug，它可能会读取代码、运行测试、定位错误、修改文件、再次运行测试，直到测试通过或者达到停止条件。

它的优点是自动化程度高，适合跑批任务、长期任务和复杂执行链路。缺点是容易跑偏，尤其当目标不清楚或者工具权限过大时，可能会产生不可控行为。所以企业落地时必须加限制，比如最大循环次数、权限边界、人工确认、日志审计和失败回滚。

整体来看，这些架构不是互相排斥的，而是可以组合使用。比如做一个企业级合同风险审查 Agent，可以用 Plan-and-Execute 做任务拆解，用 RAG Agent 检索法规和历史合同，用 Tool-Augmented Agent 调用 OCR、数据库和审批系统，用 Reflective Agent 检查审查报告质量，用 Memory-Augmented 记住企业审查规则和用户偏好。如果业务更复杂，还可以用 Multi-Agent 做角色分工。

所以我会总结为：**ReAct 解决边想边做，Plan-and-Execute 解决复杂任务拆解，Multi-Agent 解决多角色协作，Reflective Agent 解决自我纠错，Tool-Augmented 解决外部工具调用，Memory-Augmented 解决长期上下文，RAG Agent 解决知识检索和溯源，Autonomous Loop 解决持续自主执行。企业级系统里，我更倾向于用 workflow 控制主流程，用这些 Agent 架构增强关键智能节点，保证系统既智能，又可控、可观测、可回滚。**

这些概念不能简单理解成八种互斥的架构，它们更像是 Agent 系统中的不同设计模式。真实工程里，一个完整的 Agent 往往会同时包含 ReAct、规划、工具调用、RAG、记忆、反思甚至多智能体协作。

比如 ReAct 更像是 Agent 最基础的执行循环：模型先推理，再行动，再根据工具反馈继续推理。这个模式可以嵌入到很多架构里，不一定单独存在。

Plan-and-Execute 更偏向任务管理。它解决的是复杂任务怎么拆解、怎么安排执行顺序、失败后怎么重试或回滚的问题。它通常会和 ReAct 结合使用：先规划整体步骤，每个步骤内部再用 ReAct 去执行。

Tool-Augmented 是能力扩展。因为大模型本身不能真正访问数据库、运行代码、读取文件或调用业务系统，所以要通过工具调用把大模型连接到外部系统。这个能力几乎是现代 Agent 的标配，不管是 ReAct、Plan-and-Execute，还是 Multi-Agent，基本都会用到工具。

RAG Agent 解决的是知识来源问题。它让 Agent 在回答前先去知识库、文档库或向量数据库检索资料，从而减少幻觉，提高可溯源性。它本质上也常常依赖工具调用，比如向量检索工具、搜索工具、数据库查询工具。

Memory-Augmented 解决的是长期上下文问题。RAG 更偏向查外部知识，Memory 更偏向保存用户偏好、历史任务状态、项目背景和交互记录。两者可以组合，比如一个企业 Agent 既能查制度文档，也能记住当前项目进度和用户偏好。

Reflective Agent 解决的是质量控制问题。它不是一个完整系统的全部，而是一个自检机制。比如 Agent 生成代码、审查报告或解决方案后，可以再进行反思，检查是否遗漏需求、是否存在逻辑错误、是否需要修改。它经常放在执行结果之后，作为质量保障环节。

Multi-Agent 解决的是复杂任务中的角色分工问题。它可以把任务拆给多个专业 Agent，比如检索 Agent、代码 Agent、评审 Agent、报告 Agent。每个 Agent 内部仍然可能使用 ReAct、工具调用、RAG 和反思机制。所以 Multi-Agent 更像是组织方式，而不是和其他模式并列的独立类型。

Autonomous Loop 解决的是持续自主执行问题。它把“目标设定、任务拆解、执行、评估、记录、继续执行”串成一个循环。它通常会组合 Plan-and-Execute、Tool-Augmented、Memory 和 Reflective 机制。它能力最强，但风险也最大，所以企业落地时要有边界控制，比如最大循环次数、权限控制、人工确认和日志审计。

所以我会把这些概念理解成不同层面的能力：

**ReAct 是基础执行循环；Plan-and-Execute 是任务规划机制；Tool-Augmented 是外部能力接入；RAG 是知识检索增强；Memory 是长期上下文管理；Reflective 是质量自检机制；Multi-Agent 是组织协作方式；Autonomous Loop 是持续自主执行框架。**

在企业级项目里，比较合理的做法不是选择其中某一个，而是按业务需求组合。比如企业合同风险审查 Agent 可以这样设计：外层用 workflow 固定审批主流程；进入智能审查节点后，用 Plan-and-Execute 拆解任务；用 RAG 检索法规、制度和历史合同；用工具调用 OCR、数据库和审批系统；用 Reflective Agent 对风险报告做自检；用 Memory 记住企业审查规则和用户偏好；如果任务复杂，再拆成合同解析 Agent、法规检索 Agent、风险判断 Agent 和报告生成 Agent。
# 6、如果有非常多的工具，比如几万个工具，有什么优化策略，总不能全都塞进去吧?

如果一个 Agent 系统里有几千甚至几万个工具，肯定不能把所有工具 schema 一次性塞给大模型。这样会带来三个问题：**上下文太长、工具选择准确率下降、调用成本和延迟变高**。所以工程上一般会做“工具分层、工具检索、工具路由、权限收敛”。

我理解比较合理的优化策略是这样的：

**第一，工具要先分层，不要平铺。**

几万个工具不能直接作为一个大列表暴露给模型，而应该按照业务域、能力类型和权限等级进行分层管理。比如：

合同类工具、财务类工具、代码类工具、搜索类工具、数据库类工具、消息通知类工具、审批类工具。

模型第一步不需要看到所有具体工具，只需要看到一级能力目录。它先判断任务属于哪个领域，再进入对应工具集合。

比如用户问“帮我查这个合同有没有风险”，模型先进入合同审查工具域，而不是在几万个工具里盲选。

**第二，使用 Tool Retrieval，也就是工具检索。**

可以把每个工具的名称、描述、输入参数、适用场景、示例调用方式做成工具文档，然后存到向量数据库或检索系统里。

当用户提出任务时，先根据用户意图检索出最相关的 Top-K 个工具，比如 5 个、10 个或 20 个，再把这些候选工具交给大模型选择。

这样大模型看到的不是几万个工具，而是和当前任务最相关的一小批工具。这个思路和 RAG 很像，只不过检索的不是知识文档，而是工具说明。

**第三，增加 Tool Router，也就是工具路由器。**

在大模型真正调用工具之前，可以加一个路由层。这个路由层可以是规则、分类模型、小模型，或者另一个专门负责工具选择的 LLM。

它的作用是先判断任务类型，比如是查数据库、读文件、发邮件、跑代码、检索知识库，还是调用业务 API。然后只把对应类别下的工具暴露给主 Agent。

这样主 Agent 不需要承担所有工具选择压力，系统整体也更稳定。

**第四，工具 schema 要按需加载。**

很多时候模型不需要一开始就知道完整参数结构。可以先给它工具的简短描述，让它选出可能需要的工具；选中后，再加载这个工具的完整 schema。

也就是两阶段：

第一阶段：粗选工具，只看工具名称和功能摘要。  
第二阶段：精确调用，再加载完整参数定义。

这样可以明显减少上下文占用，也能降低模型被大量工具干扰的概率。

**第五，要设计工具组合或高层工具。**

几万个底层 API 不能全都直接暴露给 Agent。很多底层工具应该封装成更高层的业务工具。

比如不要让模型直接面对十几个数据库表查询接口、权限校验接口、合同解析接口和风险规则接口，而是封装成一个“合同风险审查工具”。内部怎么查库、怎么调用 OCR、怎么匹配规则，由后端服务完成。

也就是说，Agent 面向的是业务能力，不是所有底层 API。这样既降低模型选择难度，也提升安全性和可维护性。

**第六，结合权限和上下文做工具过滤。**

不是所有用户都能使用所有工具，也不是所有任务都需要所有工具。系统可以根据用户身份、当前项目、任务类型、会话上下文、数据权限，提前过滤掉不可用工具。

比如普通员工不能调用财务审批接口，外部用户不能调用内部数据库工具，代码分析任务不需要出现邮件发送工具。

这一步很重要，因为工具越多，安全风险越高。工具过滤不仅是性能优化，也是权限控制。

**第七，对工具调用结果做评估和重试。**

工具选择不是一次就一定正确。实际系统里可以记录工具调用是否成功、返回结果是否满足需求、是否需要切换工具。对于失败调用，可以让 Agent 重新选择候选工具，或者回退到路由层重新检索。

长期来看，还可以用调用日志优化工具排序：哪些工具经常被成功调用，哪些工具经常失败，哪些工具在某类任务下更有效，都可以反馈给工具检索和路由模块。

所以我会总结为：**工具很多时，核心不是把所有工具都塞给大模型，而是建立一个工具管理和路由体系。先按业务域分层，再通过检索召回候选工具，用路由器缩小范围，按需加载 schema，并把底层 API 封装成高层业务工具。同时结合权限控制、调用日志和失败重试，让 Agent 只看到当前任务真正需要的工具。**

在企业级系统里，我会更倾向于采用这种架构：**用户请求先进入意图识别和工具路由层，路由层从工具库中检索 Top-K 候选工具，再把候选工具 schema 提供给主 Agent，主 Agent 完成参数生成和调用，最后由执行器负责权限校验、真实调用、日志记录和异常处理。**这样既能解决几万个工具的规模问题，也能保证系统可控、可审计。



# 7、agent如何实现无限上下文（一直问下去不丢失之前的信息）

所谓 **Agent 的无限上下文**，并不是说真的把所有历史对话、所有文件、所有工具结果都一直塞进大模型的 context window 里。大模型本身的上下文窗口是有限的，真正工程上的“无限上下文”，本质是通过 **外部记忆系统 + 检索机制 + 摘要压缩 + 状态管理**，让模型在需要的时候取回相关信息，而不是永远不丢任何 token。

具体来说，一般会从几个层面实现。

第一层是 **短期上下文管理**。  
当前这一轮任务中，最近的对话、工具返回结果、当前执行状态会保留在上下文里。因为这些信息和当前推理最相关，比如用户刚刚提出的约束、刚调用工具得到的结果、当前执行到第几步。

第二层是 **摘要压缩**。  
当上下文越来越长时，系统不会简单丢弃旧信息，而是会把早期对话或历史执行过程压缩成摘要。比如把十几轮对话总结成：“用户目标是完成合同审查 Agent，技术栈是 Java + LangGraph，要求输出中文文档，当前已完成需求分析。”  
这样可以用更少 token 保留关键状态。

第三层是 **长期记忆**。  
对于长期有价值的信息，比如用户偏好、项目背景、历史决策、业务规则、常用工具、之前的任务结果，可以写入外部存储。这个存储可以是数据库、KV 存储、文档库，或者向量数据库。下次任务需要时，再检索出来放回上下文。

第四层是 **向量检索，也就是 Memory Retrieval**。  
系统会把历史对话、文档、工具结果切片后向量化，存入向量数据库。用户提出新问题时，先根据当前问题去检索最相关的历史片段，只把 Top-K 相关内容放进上下文。  
所以它不是“所有历史都不丢”，而是“相关信息能找回来”。

第五层是 **状态化执行**。  
对于复杂 Agent，尤其是 Plan-and-Execute 或 Autonomous Loop，系统会维护一个外部状态，比如任务列表、当前进度、已完成步骤、失败原因、下一步计划、已调用工具结果。模型每次只读取当前需要的状态，而不是依赖整个聊天历史。

第六层是 **分层记忆管理**。  
可以把记忆分成不同层级：当前会话记忆、任务级记忆、项目级记忆、长期用户记忆。不同层级有不同的保存周期和检索策略。比如当前会话记忆最详细，但生命周期短；长期记忆更稳定，但只保存高度概括的信息。

所以我会强调一点：**Agent 的无限上下文不是无限塞上下文，而是无限外部化存储、按需检索、动态压缩和状态恢复。**

举个例子，一个代码 Agent 长期维护一个项目时，它不会每次都把整个代码仓库和所有历史对话塞给模型。它会把项目结构、关键文件、接口说明、历史修改记录存到外部系统里。当前要修一个 bug 时，先检索相关文件、相关报错、历史类似问题，再把这些信息提供给模型。模型只看到当前任务最相关的上下文，但表现上像是“记得很多东西”。

工程上还要注意几个问题：

第一，记忆写入不能太随意，否则会产生大量无用信息。  
第二，检索要准确，否则取回错误上下文会误导模型。  
第三，摘要可能丢细节，所以重要信息最好结构化保存。  
第四，记忆需要更新和淘汰，否则旧信息会和新信息冲突。  
第五，涉及用户数据和企业数据时，必须做权限隔离和审计。

所以我的总结是：**大模型本身没有真正无限上下文，Agent 是通过外部 memory、RAG 检索、摘要压缩和任务状态管理来模拟无限上下文能力。真正关键不是记住所有内容，而是在正确的时间找回正确的信息。**


# 8、agent中有一个human-in-the-loop，如果让你做这个机制，你会怎么做？

Human-in-the-loop 本质上是给 Agent 加一个 **人工审批关卡**。它不是让 Agent 完全自动执行所有动作，而是在高风险操作前暂停，把操作意图、影响范围和关键参数展示给人，由人确认后再继续执行。

比如在对话过程中，Agent 判断需要删除一条数据库记录。这个时候不能让它直接删，而是进入人工确认节点：

> Agent 先生成操作计划：准备删除哪条记录、为什么删除、会影响哪些数据。  
> 系统暂停执行，并把确认请求展示给用户。  
> 用户点击“确认”后，Agent 才真正调用删除工具。  
> 删除完成后，Agent 继续后续流程，比如输出结果、更新状态、记录日志。

实现原理上，我会把它理解成 **Agent 执行流中的中断与恢复机制**。

具体流程是：

1. **Agent 运行到高风险动作**
    
    比如删除、转账、发邮件、提交审批、修改生产配置、执行 SQL、调用外部 API 等。
    
2. **系统识别这是需要人工确认的动作**
    
    可以通过工具权限等级、动作类型、业务规则来判断。比如只读工具可以自动执行，写操作、删除操作、资金操作必须人工确认。
    
3. **Agent 暂停，并保存当前状态**
    
    这里很关键。系统要保存当前任务状态，包括用户目标、当前步骤、待执行工具、工具参数、上下文摘要、执行历史等。这样用户确认后，Agent 可以从暂停点继续，而不是重新开始。
    
4. **把待执行动作展示给人确认**
    
    展示内容不能只是“是否继续”，而应该包括：
    
    - 要调用什么工具；
        
    - 具体参数是什么；
        
    - 会影响哪些对象；
        
    - 风险级别是什么；
        
    - Agent 为什么要执行这个动作。
        
5. **用户确认、拒绝或修改**
    
    用户确认后，系统恢复执行流，调用对应工具。  
    用户拒绝后，Agent 需要重新规划。  
    用户修改参数后，Agent 用新的参数继续执行。
    
6. **执行后记录审计日志**
    
    企业场景里要记录是谁确认的、什么时候确认的、执行了什么工具、参数是什么、结果是什么。这样后续可以追责和回滚。
    

所以我认为 human-in-the-loop 不是简单地在界面上弹一个确认框，而是一个完整的 **暂停、审批、恢复、审计** 机制。

如果用工作流或图执行框架来实现，我会把人工确认做成一个特殊节点。执行到这个节点时，系统把当前状态持久化，然后返回一个 pending 状态。前端展示确认页面。用户点击确认后，后端拿到确认事件，再从这个 checkpoint 恢复执行。

比如可以抽象成：

```text
用户请求
  ↓
Agent 分析任务
  ↓
生成工具调用计划
  ↓
判断是否高风险
  ↓
需要人工确认 → 暂停并保存状态 → 用户确认
  ↓
恢复执行
  ↓
调用工具
  ↓
返回结果
  ↓
继续生成最终回复
```

在工程上，我会重点设计三块：

第一是 **风险分级**。  
不是所有操作都需要人工确认。查询类、读取类工具可以自动执行；写入类、删除类、外部发送类、生产环境变更类工具必须确认。

第二是 **状态持久化**。  
Agent 暂停时要保存完整执行状态，比如当前 node、上下文、工具参数、任务 ID、用户 ID。这样确认后才能继续执行。

第三是 **权限和审计**。  
确认人必须有权限，不能谁都能点确认。同时所有确认和执行动作都要记录日志。

所以我会总结为：**human-in-the-loop 的核心实现是把高风险动作变成可中断的审批节点。Agent 在执行前生成动作计划，系统暂停并保存 checkpoint，用户确认后再恢复执行。它解决的是 Agent 自主执行中的安全性、可控性和责任边界问题。**
# 9、Harness是什么
这个将Harness分成了6层

这个分成了4层
[Harness到底是什么？四层拆解，一篇讲透AI圈最火的新概念 - 知乎](https://zhuanlan.zhihu.com/p/2024269041427072886)
> "Agent Harness" 指的是：**包在大模型外面、让它真正能"干活"的那一整套系统**。不是模型本身，而是模型以外的一切。
**Agent = 模型（Model）+ Harness**
Harness 指的是模型之外的整套系统：系统提示词、工具调用、文件系统、沙箱环境、编排逻辑、钩子中间件、反馈回路、约束机制。模型只提供推理和生成能力，Harness 把状态、工具、反馈、执行环境和安全边界串起来，Agent 才能真正开始干活。
- Model：就是 [LLM](https://zhida.zhihu.com/search?content_id=272604389&content_type=Article&match_order=1&q=LLM&zhida_source=entity)（大语言模型，比如 ChatGPT、Claude 背后的那个"大脑"），负责"想"
- Harness：围绕这个大脑搭的一切——工具、连接、环境、流程、管控，负责"干"

换句话说：

如果你只是调用一个 [API](https://zhida.zhihu.com/search?content_id=272604389&content_type=Article&match_order=1&q=API&zhida_source=entity)（应用程序接口，简单说就是让两个软件互相"对话"的通道）让模型吐字，那只是"用了个 LLM 服务"；当你给它加上文件系统、浏览器、代码执行、记忆、任务编排、监控重试……这整套东西组合在一起，才叫一个真正能"干活"的 Agent。

**这一整套"模型以外的部分"，就叫 Harness。**

###  **第一层：能力层（Skills / Tools）**
就是给 Agent 装上的各种具体技能：读写文件、操作 Excel、调浏览器、查数据库、生成摘要……每一个技能就是一个"工具"（Tool）。

### **第二层：连接层（API / mcp 等协议和接口）****
光有技能还不够，Agent 得能连上外面的服务和数据。API 接口、MCP（Model Context Protocol，模型上下文协议，可以理解为 Agent 连接工具的标准"插头"）就是这一层的核心。

### **第三层：构建层（prompt策略/ SDK / 框架 / 编排逻辑）**

这一层解决的是"Agent 怎么被搭出来、怎么组织工作"。

具体包括：

- **系统 Prompt / 策略层** —— 统一定义 Agent 的角色、风格、安全边界、输出格式。这是 Agent 的"行为准则"，告诉它"你是谁、该怎么做、什么不能做"
- **Agent SDK**（软件开发工具包）—— 提供封装好的类和函数，用来"写出一个 Agent"。比如对话循环、工具注册、消息格式这些基础零件
- **Agent 框架**（像 LangGraph、AutoGen）—— 更偏"编排框架"，用来定义多 Agent 之间的协作方式和工作流。比如一个负责写代码、一个负责测试、一个负责审查，框架决定它们怎么配合
- **任务编排逻辑** —— 子 Agent 调度、模型路由（不同任务用不同模型）、复杂工作流的执行顺序

### **第四层：运行管控层（执行环境 / 状态记忆 / 监控容错 / 评估优化）**
这一层解决的是"Agent 怎么稳定地、长期地、可控地运转"。

这一层也是 **Harness 这个概念最近火起来的核心原因**——行业发现，前三层大家都在做，但真正决定一个 Agent 产品"好不好用"的，是这一层做得够不够扎实。

具体包括：

- **执行环境（Runtime / Sandbox）** —— 比如 [Docker 沙箱](https://zhida.zhihu.com/search?content_id=272604389&content_type=Article&match_order=1&q=Docker+%E6%B2%99%E7%AE%B1&zhida_source=entity)（一个隔离的安全运行空间），让 Agent 的操作不会搞坏你的电脑或数据
- **状态与记忆** —— 长期任务状态、任务列表、外置知识库、会话日志。让 Agent 不是"每次从零开始"，而是能记住之前做了什么
- **执行控制 & Hooks / Middleware**（钩子和中间件，可以理解为在流程关键节点插入的检查和拦截机制）—— 重试、节流、监控、自动评估、打分、A/B 测试
- **可观测性** —— 把整条 Agent 轨迹结构化记录下来（每一步想了什么、调了什么工具、结果如何），支持回放调试和持续优化
总结：
- Agent = Model + Harness，Harness 就是模型以外的一切
- Harness 内部可以分四层：能力、连接、构建、运行管控
- Skills、SDK、框架、MCP 这些概念都属于 Harness 的不同层
- 最近 Harness 火起来，核心是因为大家发现运行管控层才是 Agent 能不能落地的关键瓶颈。AI产品性能差距的大头可能往往不在模型，而在 Harness设计。

# 10、Hermes Agent：会自我进化的AI助手

[Harness和Hermes分不清？看完这篇彻底懂了（附代码与架构图）_Agent Skills_AI乔见_InfoQ写作社区](https://xie.infoq.cn/article/2e54423dd9f99f7413e40cf3f)
**OpenClaw 是“能协同的万能遥控器”，而 Hermes 是“会成长的数字分身”** 。

有持久记忆、越用越懂你的 AI 助手。关键词是 **“记忆”** 。

**第一层：上下文压缩（Context Compression）**

这是一种类似“工作记忆”的机制，用于管理短时上下文。每次会话的推理过程都在这一层进行，但会话结束后不会保留。它的作用是让 Agent 在当前任务中保持连贯，而非长期存储。

**第二层：SQLite FTS5 会话搜索（SQLite FTS5 Session Search）**

这一层负责检索历史会话的“冷存储”。所有 CLI 和消息渠道的会话记录都存入 SQLite 数据库，Agent 可以通过 `session_search` 工具主动检索——“我们上周讨论过认证服务吗？”这种跨会话的 episodic recall 完全依赖这一层。

**第三层：持久化的 MEMORY.md（Persistent MEMORY.md）**

这一层存储跨会话的核心事实与知识，类似人脑中的海马体，负责将重要信息固化为长期记忆。`MEMORY.md`（约 2,200 字符）存储持久事实，`USER.md`（约 1,375 字符）存储用户画像。两者在每次会话开始时作为“冻结快照”加载到上下文中，确保 Agent 始终知道你告诉过它的关键信息。

更重要的是，Hermes 内置了 **“提醒间隔”（nudge_interval）机制**——在会话运行的固定间隔里，Agent 会收到一条内部系统提示，主动复盘当前会话中值得保存的内容，不需要用户插手。

可以说，OpenClaw 的记忆系统是“被动响应式”的，而 Hermes 的记忆系统是“主动构建式”的。


技能系统是 Hermes 最具差异化的功能，但技能不是凭空产生的——驱动这一切的，是 Hermes 底层的一套**闭环学习循环**。它不是某个独立的功能模块，而是一个贯穿每次会话的闭合反馈系统，由四个核心机制共同咬合驱动。

整个循环的运转由四个齿轮咬合驱动：

**齿轮一：周期性 Nudge——“该学习了”的闹钟**

这是最底层的触发机制。每执行 10 个 turn，系统就自动给 Agent 发一条“该学习了”的信号。这背后的逻辑是：用户在对话过程中产生的隐式知识——比如反复修改某个需求、某个解释方式特别有效、用户对某类问题特别敏感——这些信息如果不主动触发，Agent 不会自己想起来要记。Nudge 把“学习”从用户的负担变成了 Agent 的本能——你不用专门说“帮我记住这个”，系统会提醒它该整理了。

**齿轮二：后台 Background Review——不阻塞主进程的异步复盘**

这是 Hermes 工程上最亮眼的设计之一。传统的实现方式是：Agent 每次对话结束，立即开始复盘、总结、存入记忆，这个过程是同步的——用户在那等着，Agent 在后台吭哧吭哧整理。如果用户短时间内发来新消息，主进程就会被阻塞。Hermes 的做法是：fork 一个独立的 Agent 专门做 Background Review，以 daemon 线程的方式运行，完全不阻塞主对话流程。

**齿轮三：自主精选记忆——主动筛选，不堆垃圾**

大多数 Agent 要么啥都记、啥都用不上，要么啥都不记、每次从零开始。Hermes 把筛选记忆的权力交给了自己：在会话运行的固定间隔里，Agent 会收到一条内部系统提示，主动复盘刚才发生的事儿里有没有值得存下来的。整个过程不需要用户插手，Agent 自己扫一遍近期操作，觉得对未来有用就写进记忆文件。结果就是：记忆库始终是精选内容，而不是乱七八糟的聊天垃圾堆。

**齿轮四：自主生成技能——从经验到方法论**

Agent 干完一个任务后，会自动判断：我刚才这套操作流程，值不值得写成技能文档？触发条件非常明确：调用了 5 次及以上工具、从错误里成功恢复、用户给过修正指导、走通了一套不那么直观的有效流程。满足条件后，Agent 就会把经验固化为技能。

自动生成产生的技能文件（SKILL.md）不是普通日志，而是一套下次能直接照搬的可复用指令集。每个技能文件包含：技能名称和描述、版本号和适用平台、元数据标签、**完整的执行步骤**（下次遇到同类任务可以直接按步骤执行，无需从头推理），以及用到的工具调用和文件引用。

技能文件格式完全遵循 **agentskills.io** 开放标准，意味着这些技能可以在兼容的 Agent 之间迁移。更值得关注的是**技能自我改进**：当 Agent 在后续任务中调用某个技能时，如果发现这个技能已经过时、不完整或有错误，它会主动进行修复——使用精准的 find-and-replace 方式，只修改需要变更的片段，而不是重写整个文档。这意味着技能会随着使用持续优化，越用越准。
