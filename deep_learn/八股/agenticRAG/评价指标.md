可以分成三类：

```
1. 公认通用指标
2. RAG/LLM 评测中常见但本项目用 LLM Judge 实现的指标
3. 本项目为多跳 Agent / GRPO reward 自定义或改造的诊断指标
```

**一、公认通用指标**

这些是最标准、最容易被面试官接受的。

`EM / Exact Match`

含义：

```
预测答案和标准答案完全匹配就是 1，否则就是 0
```

项目里会先做 normalize：

```
转小写
去英文标点
去英文冠词 a/an/the
合并空格
```

然后比较：

```
normalize(pred) == normalize(gold)
```

支持 `answer_aliases`，只要和任意标准答案别名完全匹配，EM 就是 1。

意义：

```
衡量答案是否严格命中标准答案
```

缺点：

```
太严格，中文金融 QA 里容易低估模型表现
```

---

`F1 / Token F1`

含义：

```
预测答案和标准答案的 token 重叠程度
```

计算：

```
precision = 重合 token 数 / 预测答案 token 数
recall    = 重合 token 数 / 标准答案 token 数
F1        = 2 * precision * recall / (precision + recall)
```

同样支持 `answer_aliases`，取最高 F1。

意义：

```
比 EM 宽松，能衡量部分匹配
```

缺点：

```
中文分词粒度不稳定
comparison 问题中，模型多说几个字就可能被扣很多分
```

比如 gold 是：

```
永辉超市
```

模型答：

```
永辉超市的营业收入更高
```

语义是对的，但 F1 可能不高。

---

**二、RAG/LLM 评测中常见，但本项目用 LLM Judge 实现的指标**

这些指标不是本项目首创，RAG 领域经常用，但具体实现是项目自己写 prompt 让 `gpt-oss-120b` 判分。

`Judge_C / Correctness`

`Judge_C` 是：

```
Judge Correctness
```

含义：

```
让 LLM Judge 判断模型答案和 gold answer 是否语义等价
```

它不是简单字符串匹配，而是语义判断。

高了说明：

```
最终答案更正确
```

和 EM/F1 的区别：

```
EM/F1 是规则匹配
Judge_C 是语义级判断
```

---

`Faith / Faithfulness`

含义：

```
答案是否被检索到的 evidence 支持
```

它检查的是：

```
模型是不是根据证据回答
还是脱离 evidence 靠猜
```

高了说明：

```
答案更有依据，更少幻觉
```

注意它和 Correctness 不一样：

```
Correctness：答案对不对
Faithfulness：答案是不是证据支持的
```

可能出现：

```
答案对，但 evidence 不支持 → Correctness 高，Faithfulness 低
答案有证据，但抽取/推理错了 → Faithfulness 可能高，Correctness 低
```

---

`CtxP / Context Precision`

`CtxP` 是：

```
Context Precision
```

含义：

```
检索到的 evidence 是否精准、相关、少噪声
```

可以通俗理解成：

```
搜回来的内容里，有多少是真的有用证据
```

高了说明：

```
检索更精准，返回的无关 chunk 更少
```

它和 `hop_recall` 不一样：

```
CtxP：看证据是否精准相关，通常由 LLM Judge 评估
hop_recall：看 gold chunk_id 有没有被命中，是硬匹配
```

---

**三、本项目自定义 / 改造的多跳 Agent 指标**

这些更偏项目特色，用来诊断 Agent 在多跳检索中哪里失败。

`hop_recall`

这个是项目里的 hop-aware 诊断指标，来源思想借鉴 AgenticRAGTracer，但在本项目里按 `chunk_id` 实现。

含义：

```
gold hops 需要的标准证据 chunk，有多少被模型检索到了
```

计算：

```
gold_chunk_ids = 所有 hop.doc_chunk_id
retrieved_ids  = Agent 检索到的所有 chunk_id

hop_recall = |gold_chunk_ids ∩ retrieved_ids| / |gold_chunk_ids|
```

例子：

```
标准需要 3 个 chunk
模型搜到其中 2 个
hop_recall = 2 / 3
```

意义：

```
判断模型有没有把多跳证据链找全
```

---

`hop_precision_recall`

这是 GRPO reward 里更进一步的检索质量指标。

`hop_recall` 只看找没找全，不惩罚乱搜。

如果标准需要：

```
{A, B}
```

模型搜到：

```
{A, B, C, D, E}
```

`hop_recall = 1.0`，但其实噪声很多。

所以项目用了 `hop_precision_recall`，本质是 gold chunk 命中的 F1：

```
hit = |gold_chunks ∩ retrieved_chunks|

recall = hit / |gold_chunks|
precision = hit / |retrieved_chunks|

hop_precision_recall = 2 * precision * recall / (precision + recall)
```

意义：

```
同时奖励“找全”和“少搜噪声”
```

---

`premature_collapse`

含义：

```
Agent 过早停止，还没搜够证据就回答
```

项目里的判断逻辑大致是：

```
hop_recall < 0.5 且 iteration_count <= 1
```

高了说明：

```
模型容易没搜完整就提前给答案
```

---

`over_extension`

含义：

```
Agent 搜太多、查太久、扩展过度
```

项目里的判断逻辑：

```
total_tool_calls > hop_count * 3
```

高了说明：

```
模型检索失控，可能反复搜索或引入噪声
```

---

`step_alignment`

含义：

```
Agent 实际执行的证据步骤数和 gold hop 数是否匹配
```

计算：

```
step_alignment = evidence_count / hop_count
```

意义：

```
看 Agent 的执行步数和问题真实 hop 数是否大致对齐
```

---

`Grounded Answer`

这是 GRPO reward 里的辅助规则指标。

含义：

```
最终答案里的 token，有多少出现在 evidence 里
```

计算：

```
grounded_answer = |answer_tokens ∩ evidence_tokens| / |answer_tokens|
```

意义：

```
鼓励模型从检索证据中抽取答案，而不是凭空生成
```

它不是严格的 faithfulness，只是一个 token overlap 近似指标。

---

`Format Reward`

这是本项目训练时的格式奖励。

含义：

```
模型是否输出合法 tool_call 和 <answer> 标签
```

例如奖励：

```
有 <answer>...</answer>
有合法 <tool_call>{...}</tool_call>
JSON 可解析
工具名和参数格式正确
```

意义：

```
让模型先学会可执行的 Agent 协议
```

这个是训练 reward，不是最终核心评测指标。

---

**怎么面试总结**

可以这样说：

> 项目里的指标分三类。第一类是公认的 QA 指标 EM 和 Token F1，分别衡量严格匹配和 token 级部分匹配。第二类是 RAG/LLM 评测常见指标，包括 Correctness、Faithfulness 和 Context Precision，本项目用 gpt-oss-120b 做 LLM Judge 实现，分别判断答案是否正确、是否被证据支持、检索上下文是否精准。第三类是多跳 Agent 诊断指标，主要是 hop_recall、hop_precision_recall、premature_collapse、over_extension 和 step_alignment，其中 hop_recall 看 gold chunk 是否找全，hop_precision_recall 同时考虑找全和少噪声。这些指标帮助定位模型到底是答案生成错、证据没搜全、搜太多，还是过早停止。
