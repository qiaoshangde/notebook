**简介**
我做了一个 **面向中文金融场景的多跳 Agentic RAG 项目**。这个项目的目标是解决金融财报问答里常见的复杂多跳问题，也就是一个问题不能靠一次检索直接回答，而是需要跨多个文档、多个证据块逐步检索和推理。

比如用户问：

```
永辉超市和红旗连锁哪家公司某项财务指标更高？更高的这家公司法人代表是谁？
```

这种问题就不是单轮 RAG 能很好解决的。系统需要先分别检索两家公司的财务指标，比较出结果，然后再根据比较结果继续检索对应公司的法人代表，最后才能回答。所以这个项目本质上是在做一个能自主多轮搜索、证据收集和推理的金融问答 Agent。

整个项目我主要做了四块工作：

```
第一，构建金融多跳 QA 数据集
第二，设计 Agentic RAG 检索与推理架构
第三，用 SFT 训练小模型学会工具调用
第四，用 GRPO 强化学习提升小模型的自主搜索和回答能力
```


**知识库**
先说数据侧。金融领域没有现成的高质量中文多跳 QA 数据，所以我从原始财报开始构建数据集。原始语料来自多家零售上市公司的 2025 年半年度报告 PDF。我用 PyMuPDF 做页面文本解析，然后按段落切成 chunk，每个 chunk 大约 500 字符，并保留一定 overlap，最终形成可检索的金融 corpus（语料库）。

在 corpus 之上，我构建了三类检索索引。第一类是基于 BGE-M3 的向量索引，用 FAISS 做语义检索；第二类是 BM25 关键词索引，主要处理公司名、财务科目、日期、金额这类精确匹配问题(中英文切分与A股份等)；第三类是知识图谱检索，用 LLM 从 chunk 中抽取实体关系三元组，再用图遍历找到跨实体、跨文档的关联证据。后面 Agent 可以调用 `keyword_search`、`semantic_search`、`graph_search` 和 `hybrid_search` 等工具。

数据合成部分，我参考了 AgenticRAGTracer 的 hop-aware 数据合成思路，但把语料和 prompt 都适配到了中文金融领域。具体流程是先让 LLM 从每个 chunk 中提取原子化的单跳 seed QA，比如“某家公司 2025 年上半年经营活动现金流净额是多少”。然后通过逐跳扩展，把多个 seed QA 合成为 2-hop、3-hop、4-hop 的多跳问题。数据里会保留完整的 `hops` 结构，也就是每一步的子问题、中间答案和对应的 gold chunk。这个 `hops` 结构后面非常重要，它既能用于评测，也能用于构造 SFT 的 oracle trajectory。

**训练数据合成**
首先，我用 7 家中国零售上市公司的半年报 PDF 作为原始数据源。PDF 先用 PyMuPDF 解析成文本，然后按段落切成 chunk，每个 chunk 大约 500 字符，并保留 50 字符左右的重叠，避免切分时把关键信息截断。最终得到 1933 个 corpus chunks。这里的 chunk 就是后续检索系统的最小证据单元。

接着，从这些 chunk 里生成 seed QA，也就是单跳的原子问答。做法是让 LLM 针对每个 chunk 提取一些事实型问题，比如某个公司某项财务指标是多少、某个时间发生了什么。生成后会做基础过滤，比如答案不能太长，问题里不能直接泄露答案，不能是重复问题，也不能是需要多个答案拼在一起的问题。这个阶段一共生成了 7437 条 seed QA，后续实际分批使用了大约 4000 条。

然后，从 seed QA 出发做多跳扩展。系统会把上一跳的问题或答案作为线索，用关键词检索、语义检索、图检索去 corpus 里找相关 chunk，再基于新 chunk 生成下一跳 QA，最后把多跳链合并成一个最终问题。比如第一跳先找到某家公司，第二跳再查这家公司某个财务指标，第三跳再和另一家公司做比较。这样就形成了带有 hop 结构的多跳 QA。

在多跳合成过程中，会先做四重验证。第一，检查多跳问题的语义逻辑是否成立；第二，不给任何文档时 LLM 不应该答对，防止它变成常识题或答案泄露题；第三，只给部分文档时 LLM 不应该答对，通常是给 N-1 个 gold chunks，保证每一跳证据都是必要的；第四，给全部 gold chunks 时 LLM 必须能答对，保证这个问题确实可以由证据回答。只有通过四重验证的样本，才会进入 raw multi-hop QA。这个阶段最终得到 2212 条 raw multi-hop QA。

之后再做规则清洗。这里主要过滤几类明显质量不好的样本：第一，过滤 trivial 最后一跳，比如 3hop 或 4hop 最后一跳只是“期末余额是多少”这种简单查表；第二，过滤所有 hop 都很 trivial 的问题，避免多跳链只是几个简单查表拼起来；第三，按问题前缀去重，避免问题表达高度重复；第四，按 chunk_id 重叠去重，如果两条 QA 依赖的证据 chunk 重叠太高，就只保留一条。规则清洗后大约剩下 1708 条。

接着用 LLM Judge 做进一步质量评分。Judge 会看完整 hop 链和对应证据，从答案正确性、多跳必要性、问题清晰度等维度打分。2hop 和 3hop 使用较严格的阈值，4hop 因为生成成本高、数量少，阈值适当放宽。经过 Judge 过滤后，得到 1340 条高质量多跳 QA，其中 2hop 有 886 条，3hop 有 347 条，4hop 有 107 条。

然后做训测划分。这里不是简单随机切分，而是按 seed chunk 分组做 80/20 划分，保证同一个种子 chunk 生成的问题不会同时出现在训练集和测试集里，避免数据泄露。划分后训练集 1065 条，测试集 275 条。

最后做搜索可达性过滤。因为这个项目后面要训练 Agent 调工具检索，所以必须保证每一跳的 gold chunk 理论上能被现有检索工具找到。做法是对每条 QA 的每一跳，用 keyword search、semantic search、graph search 以及它们的融合组合去检索，看 gold chunk 是否能出现在 top-10 里。只要某一跳完全搜不到，这条样本就会被去掉。过滤后最终保留 797 条训练集和 185 条测试集。

**简短表述：**
'''
为了保证数据质量，我做了多层过滤。首先是四重验证：第一，检查多跳问题的语义逻辑是否成立；第二，不给任何文档时 LLM 不应该答对，避免常识题或答案泄露；第三，只给 N-1 个文档时 LLM 不应该答对，保证每一跳证据都是必要的；第四，给全部 gold chunks 时 LLM 必须能答对，保证问题本身可回答。之后再做规则清洗，比如过滤 trivial 最后一跳、过滤所有 hop 都是简单查表的问题、问题前缀去重、chunk 重叠去重。最后再用 LLM Judge 从答案正确性、多跳必要性、问题清晰度三个维度打分。最终还做了搜索可达性过滤，确保每一跳的 gold chunk 能被现有检索工具 top-k 找到，否则这条样本对 SFT 和 GRPO 都会产生噪声。
'''
最终数据漏斗大概是：从约 4000 条 seed QA 出发，合成 2212 条原始多跳 QA，规则清洗后约 1708 条，LLM Judge 后保留 1340 条高质量样本，再划分训练和测试，经过搜索可达性过滤后，最终训练集 797 条、测试集 185 条。


第二块是 Agentic RAG 架构。项目里我实现了一个基于 LangGraph 的 PEV 循环，也就是 Plan、Execute、Verify。它不是简单的一次检索一次回答，而是把多跳问答拆成多个节点：

```
Router 判断问题类型
Planner 规划子问题和工具
Executor 执行检索
Verifier 判断证据是否足够
Synthesizer 基于证据生成最终答案
```

如果 Verifier 判断证据不足，会把反馈交给 Planner 重新规划。这个过程可以循环多轮。为了找到更好的架构，我在 AgenticRAGTracer benchmark 和金融测试集上做了多轮消融和迭代，比如是否需要 Verifier、是否需要 replan、不同工具组合效果如何、hybrid search 是否有效。最终比较有效的是带 Verifier 和 hybrid search 的版本。这个阶段的重点是验证 Agentic RAG 架构本身在多跳任务里的有效性。

在完成 Agentic RAG 架构探索之后，我发现一个问题：如果直接用通用大模型，比如 GPT 类模型，Pipeline 的规划、检索、验证能力是可以跑起来的，但这并不适合落到一个可控、低成本、可私有化部署的垂直领域系统里。金融场景对数据隐私、调用成本和稳定性要求比较高，如果每次多跳问答都依赖外部大模型 API，成本和延迟都会比较高，而且数据也不一定方便出域。因此我进一步考虑能不能把这套 Agentic RAG 能力迁移到一个本地小模型上，比如 Qwen3-4B，让它自己学会在金融问答中调用检索工具、根据证据回答问题。

但是小模型直接做 Agentic RAG 会有明显短板。它不是简单地缺金融知识，而是不会稳定地按照 Agent 的方式工作：比如不会规范地产生 tool call，不知道什么时候该继续搜索，检索 query 写得不准，拿到证据后也容易脱离 evidence 直接猜答案。所以后训练的目标不是让模型单纯“记住金融知识”，而是让它学会一套垂直领域 Agent 行为模式：先通过 SFT 学会标准的工具调用格式和基本搜索轨迹，再通过 GRPO 强化它在多轮检索中选择更有效证据、基于证据生成答案的能力。

这里没有选择直接去后训练 Pipeline 里的 Planner、Verifier 等单个组件，主要是因为 Pipeline 架构本身是由代码强控制的流程：Planner 只负责拆计划，Executor 只负责调工具，Verifier 只负责判断证据够不够，每个模块的输入输出都比较固定。如果分别训练这些组件，就需要为每个模块单独构造监督数据和奖励函数，比如 Planner 要有标准计划，Verifier 要有证据充分性的标签，Executor 要有工具选择和检索命中的奖励，训练链路会被拆得很碎，而且误差会在模块之间传递。某个模块单独看指标变好了，不一定代表整个 Agent 最终回答变好。

所以这个项目更偏向训练一个端到端的 Agentic 小模型，让模型在统一的多轮交互环境里学习“什么时候搜索、搜什么、是否继续搜、最后如何基于证据回答”。这样 SFT 和 GRPO 的训练目标可以直接围绕最终任务设计，也就是多跳金融问答里的证据命中、答案正确性和忠实性。换句话说，Pipeline 里的 Planner、Verifier 更像是前期架构探索和数据生成、验证的工具，而后训练阶段关注的是把这些能力压缩进一个可部署的小模型 Agent，而不是把固定 Pipeline 的每个组件都单独训练一遍。


第三块是 SFT。因为直接让 Qwen3-4B 这种小模型做多轮 tool calling，效果非常差。基础模型虽然有一定语言能力，但不稳定地生成合法 tool call，也不知道什么时候继续搜索、什么时候回答。所以我用前面数据里的 gold hops 构造了 oracle trajectory。

所谓 oracle trajectory，就是根据标准 hops 人工构造一条理想工具调用轨迹。比如第一跳应该查什么 query，应该命中哪个 gold chunk，第二跳应该查什么，最后应该输出什么答案。SFT 数据不是简单的“问题到答案”，而是完整的 ReAct 格式多轮对话：

```
用户问题
模型 think
模型 tool_call
工具返回 tool_response
模型继续 tool_call
最后输出 <answer>
```

SFT 的目的不是主要灌金融知识，而是教模型工具调用协议和基本搜索流程。包括 tool call JSON 格式、query 构造、读取 tool response、继续搜索和最终答案格式。这里我也处理了 SFT 和 GRPO 格式对齐的问题，比如使用 hermes 格式的 `name/arguments`，最后答案加 `<answer>` 标签，并用 `apply_chat_template(tools=...)` 注入工具定义，保证 SFT 和 GRPO rollout 的 tokenization 一致。

SFT 使用 Qwen3-4B + LoRA，通过 LLaMA-Factory 训练。训练 3 个 epoch 后发现第 2 个 epoch 的 checkpoint 泛化最好，第 3 个 epoch 虽然 training loss 更低，但 held-out 指标下降，说明开始过拟合，所以后续 GRPO 以 epoch 2 的 checkpoint 作为基线。

第四块是 GRPO。SFT 只能教模型模仿理想轨迹，但真实 Agentic 场景里模型会搜错、重复搜、提前停止，SFT 的 oracle 轨迹覆盖不了这些情况。所以后面我用 verl 做 multi-turn Agentic GRPO，让模型在真实检索环境里自己 rollout。GRPO 数据里不再提供 oracle evidence，只给 system prompt、用户问题、标准答案、answer aliases、gold chunks 和 hop count。模型需要自己决定搜什么、调用哪个工具、搜几轮、什么时候停止，然后 reward 函数根据最终行为打分。

GRPO reward 设计是项目里最核心的部分之一。我尝试过多轮 reward 设计，最后总结出一个重要现象：Correctness 和 Faithfulness 存在跷跷板。如果 reward 过度强调最终答案正确，模型容易绕过 evidence，靠猜或者参数记忆回答；如果过度强调 evidence grounding，模型更依赖证据，但最终答案质量可能不够好。

所以我设计了两阶段 curriculum learning。第一阶段是 grounding-first，主要让模型形成“先搜索、拿证据、基于证据回答”的行为模式，避免直接编答案。第二阶段再切到 correctness-heavy reward，让模型在已经建立 evidence grounding 的基础上提升最终答案正确性。这个训练顺序比单阶段直接训 correctness 更稳定，因为它先缩小了策略空间，排除了“不搜索直接猜”这种 reward hacking 路径。

评测方面，我没有只看 EM/F1。因为中文金融 QA 里答案形式很多，比如 gold 是“永辉超市”，模型答“永辉超市的营业收入更高”语义是对的，但 token F1 可能不高。所以我用了多维度评测：

```
EM / F1：传统答案匹配
Judge_C：LLM Judge 判断答案正确性
Faithfulness：答案是否被证据支持
Context Precision：检索证据是否精准
hop_recall：标准 gold chunks 是否被找全
premature_collapse：是否过早停止
over_extension：是否过度检索
```

最终结果上，纯 SFT 已经能让 Qwen3-4B 从不会自主 tool calling 提升到可用水平；再经过两阶段 GRPO 后，模型在 Agentic 模式下的 Judge_C 有进一步提升，同时 Faithfulness 也比直接 correctness-heavy 的训练方案更好。这个结果说明，小模型并不是天然会做多轮 Agentic RAG，但可以通过 SFT 学会工具调用格式，再通过 GRPO 在真实环境中优化搜索和回答策略。

这个项目最大的收获是我完整跑通了一个垂直领域 Agentic RAG 的闭环：从 PDF 文档解析、语料切块、索引构建、多跳 QA 合成、数据清洗、Agent 架构设计、SFT 微调，到 GRPO 强化学习和多维度评测。同时也踩了很多工程坑，比如 SFT 和 GRPO 的格式对齐、LLM Judge reward 和评测 Judge 的分布偏移、检索可达性过滤带来的 selection bias、GRPO 显存与 vLLM rollout 的资源冲突等。

如果要继续改进，我认为有三个方向。第一是扩大金融语料规模，现在只有几家公司的半年报，生产级系统应该扩展到更多公司和年份。第二是改进检索困难样本的处理，现在可达性过滤会让训练集偏向容易检索的问题，后续可以加入搜索失败恢复轨迹。第三是把两阶段训练进一步细化成三阶段：先专门训检索命中，再训基于证据回答，最后训最终 correctness。这样可能进一步缓解检索质量、Faithfulness 和 Correctness 之间的冲突。
